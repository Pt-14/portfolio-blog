3:T1f8a,<h2>Vấn đề mà mọi lập trình viên web đều gặp phải</h2>
<p>Bạn đã bao giờ tự hỏi: Làm thế nào để ứng dụng web có thể "nói chuyện" với server? Tại sao khi click "Đăng nhập" thì dữ liệu lại được gửi đi và nhận về phản hồi? Đó chính là lúc chúng ta cần đến HTTP requests - cầu nối giữa frontend và backend.</p>
<p>Trong quá trình học JavaScript, tôi đã vật lộn với XMLHttpRequest - một API cũ kỹ với cú pháp phức tạp, callback hell, và khó debug. Fetch API ra đời như một làn gió mới, giải quyết tất cả những vấn đề đó. Bài viết này sẽ chia sẻ hành trình của tôi từ XMLHttpRequest đến Fetch API, và tại sao bạn cũng nên học nó.</p>
<h2>Fetch API là gì và tại sao nó quan trọng?</h2>
<p>Fetch API là một interface hiện đại của JavaScript, được thiết kế để thay thế XMLHttpRequest. Nó cung cấp cách thức đơn giản, mạnh mẽ để thực hiện các HTTP requests từ browser đến server.</p>
<p>Theo góc nhìn của tôi, Fetch API quan trọng vì nó đơn giản hóa việc xử lý bất đồng bộ. Từ những ngày phải vật lộn với callback hell của XMLHttpRequest, giờ tôi có thể viết code async một cách tự nhiên với async/await. Tôi nhớ lần đầu tiên dùng Fetch API trong dự án thực tế. Thay vì viết hàng chục dòng XMLHttpRequest phức tạp, chỉ cần vài dòng code là xong. Đó là khoảnh khắc "aha" khiến tôi yêu thích JavaScript hơn.</p>
<h2>Hiểu về GET Request - "Xin" dữ liệu từ server</h2>
<p>GET là request đơn giản nhất - bạn chỉ muốn lấy dữ liệu từ server. Trong thực tế, đây là request phổ biến nhất: lấy danh sách sản phẩm, thông tin user, hay dữ liệu từ API.</p>
<p>Cách tôi hiểu GET request rất đơn giản: Như việc bạn hỏi "Server ơi, cho tôi xem danh sách users nhé?" Server sẽ trả về dữ liệu mà không thay đổi gì. Hoàn toàn an toàn, không gây tác dụng phụ.</p>
<p>Trong dự án đầu tiên của tôi, tôi dùng GET để lấy danh sách sản phẩm từ API. Code đơn giản đến bất ngờ:</p>
<pre><code class="language-javascript">async function getUsers() {
  const response = await fetch('https://jsonplaceholder.typicode.com/users');
  const users = await response.json();
  return users;
}
</code></pre>
<p>Đơn giản phải không? Chỉ vài dòng code là lấy được dữ liệu từ API. Đây là lý do tôi yêu thích Fetch API.</p>
<pre><code>
### POST Request - Gửi dữ liệu

Gửi dữ liệu lên server với POST request:

```javascript
async function createUser(userData) {
  const response = await fetch('https://api.example.com/users', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(userData)
  });

  const newUser = await response.json();
  return newUser;
}
</code></pre>
<h2>Các phương thức HTTP khác</h2>
<p>Ngoài GET và POST, còn có PUT (cập nhật), PATCH (cập nhật một phần), DELETE (xóa). Những phương thức này tạo thành bộ CRUD hoàn chỉnh cho REST APIs.</p>
<p>Trong thực tế, tôi thường dùng POST và GET nhiều nhất. PUT/PATCH/DELETE ít dùng hơn vì hầu hết ứng dụng web tập trung vào việc tạo và xem dữ liệu hơn là chỉnh sửa/xóa.</p>
<pre><code class="language-javascript">// PUT - Cập nhật toàn bộ
async function updateUser(userId, userData) {
  const response = await fetch(`https://api.example.com/users/${userId}`, {
    method: 'PUT',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(userData)
  });
  return response.json();
}

// DELETE - Xóa
async function deleteUser(userId) {
  const response = await fetch(`https://api.example.com/users/${userId}`, {
    method: 'DELETE'
  });
  return response.ok;
}
</code></pre>
<h2>Xử lý lỗi</h2>
<p>Một trong những "điểm đau" lớn nhất khi làm việc với APIs là xử lý lỗi. Tôi từng có dự án bị crash vì không handle lỗi properly.</p>
<p>Bài học từ kinh nghiệm: Fetch không tự động reject với HTTP error codes (404, 500...). Phải chủ động check <code>response.ok</code> trước khi xử lý dữ liệu.</p>
<pre><code class="language-javascript">async function getData() {
  const response = await fetch('/api/data');

  if (!response.ok) {
    throw new Error(`HTTP ${response.status}`);
  }

  return response.json();
}
</code></pre>
<h2>Authentication và Headers</h2>
<p>Headers không chỉ để chỉ định content-type, mà còn quan trọng cho bảo mật. Trong dự án thực tế, hầu hết APIs đều cần authentication.</p>
<p>JWT Bearer Token là phổ biến nhất:</p>
<pre><code class="language-javascript">async function getProtectedData(token) {
  const response = await fetch('/api/protected', {
    headers: { 'Authorization': `Bearer ${token}` }
  });

  if (!response.ok) {
    throw new Error('Không thể lấy dữ liệu bảo mật');
  }

  return response.json();
}
</code></pre>
<h2>Bài học từ kinh nghiệm thực tế</h2>
<p>Sau khi áp dụng Fetch API trong nhiều dự án, tôi rút ra một số bài học quan trọng.</p>
<p>Thứ nhất, error handling là yếu tố sống còn. Tôi từng có dự án bị crash vì không check response.ok. Bây giờ tôi luôn kiểm tra status code trước khi xử lý dữ liệu.</p>
<p>Thứ hai, Fetch API đã thay đổi cách tôi viết JavaScript. Từ việc sợ hãi XMLHttpRequest, giờ tôi có thể viết network requests một cách tự tin. Code ngắn gọn, dễ đọc, và dễ maintain.</p>
<p>Thứ ba, trong thực tế, GET và POST là hai phương thức tôi dùng nhiều nhất. Các phương thức khác như PUT, DELETE ít dùng hơn vì focus vào việc tạo và xem dữ liệu.</p>
<h2>Kết luận: Fetch API - Cầu nối giữa frontend và backend</h2>
<p>Quay lại câu hỏi đầu bài: "Làm thế nào ứng dụng web 'nói chuyện' với server?" - Fetch API chính là câu trả lời hiện đại và mạnh mẽ nhất.</p>
<p><strong>Tóm tắt những gì chúng ta đã học:</strong></p>
<ol>
<li><strong>GET</strong> - Lấy dữ liệu từ server</li>
<li><strong>POST</strong> - Tạo dữ liệu mới</li>
<li><strong>PUT/PATCH/DELETE</strong> - Hoàn thiện CRUD operations</li>
<li><strong>Error handling</strong> - Quan trọng không kém code chính</li>
<li><strong>Authentication</strong> - Bảo mật là ưu tiên số 1</li>
<li><strong>Performance</strong> - Timeout, caching, parallel requests</li>
<li><strong>File operations</strong> - Upload/download trong thực tế</li>
</ol>
