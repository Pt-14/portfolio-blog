3:T26f1,<h2>Khi mình "đấu tranh" với địa chỉ mạng trong Java</h2>
<p>Xin chào các bạn! Hôm nay mình muốn chia sẻ về hành trình học Network Addresses trong Java. Đây là chủ đề mà lúc đầu mình thấy boring nhất - "địa chỉ IP, hostname, URL gì mà khó thế?"</p>
<p>Nhưng khi làm network programming thực tế, mình mới vỡ lẽ tầm quan trọng. Client connect đến server bằng IP hay hostname? URL validation? DNS lookup? - tất cả đều cần handle properly.</p>
<p>Mình còn nhớ project đầu tiên: simple client-server app. Mình hardcode IP address, nghĩ vậy là đủ. Nhưng khi deploy, ôi trời ơi:</p>
<ul>
<li>Localhost work, nhưng trên server thì không</li>
<li>DNS resolution fail</li>
<li>Invalid URLs crash app</li>
<li>IPv4/IPv6 compatibility issues</li>
</ul>
<p>Đó là lúc mình học được InetAddress, URL, URI classes. Từ confuse với network addresses, giờ mình có thể handle chúng proficiently.</p>
<p>Bài viết này là những gì mình học được sau 2 tháng vật lộn với network addresses trong Java.</p>
<h2>InetAddress - Khi mình học cách "dịch" hostname thành IP</h2>
<p>Lúc đầu, mình confuse với khái niệm hostname vs IP address. "google.com là hostname, 8.8.8.8 là IP" - OK, nhưng làm thế nào để Java convert giữa 2 thứ này?</p>
<p><strong>DNS Lookup cơ bản:</strong>
Mình bắt đầu với code đơn giản nhất:</p>
<pre><code class="language-java">import java.net.InetAddress;

public class DnsLookup {
    public static void main(String[] args) {
        try {
            InetAddress address = InetAddress.getByName("google.com");
            System.out.println("IP của google.com: " + address.getHostAddress());
            System.out.println("Hostname: " + address.getHostName());
        } catch (Exception e) {
            System.err.println("DNS lookup failed: " + e.getMessage());
        }
    }
}
</code></pre>
<p>Wow! Chỉ 1 dòng code <code>getByName()</code> là có thể resolve hostname thành IP. Mình đã thử với nhiều domains: localhost, google.com, github.com - tất cả work.</p>
<p><strong>Reverse DNS lookup:</strong>
Mình cũng thử ngược lại - từ IP ra hostname:</p>
<pre><code class="language-java">InetAddress address = InetAddress.getByName("8.8.8.8");
System.out.println("Hostname của 8.8.8.8: " + address.getHostName());
</code></pre>
<p>Kết quả: "dns.google" - interesting!</p>
<p><strong>So với system commands:</strong>
Trước khi học Java, mình dùng <code>nslookup</code> command. Nhưng Java có lợi thế:</p>
<ul>
<li>Cross-platform (Windows/Linux/macOS)</li>
<li>Exception handling tốt</li>
<li>Integrate mượt với Java code</li>
<li>No need parse command output</li>
</ul>
<p>Trong monitoring app của mình, InetAddress thay thế hoàn toàn ping commands. Reliable hơn và easy to automate.</p>
<h3>Những gì mình khám phá thêm về InetAddress</h3>
<p><strong>Multiple IP addresses:</strong>
Một hostname có thể có nhiều IP addresses (load balancing, redundancy). Mình học được <code>getAllByName()</code>:</p>
<pre><code class="language-java">InetAddress[] addresses = InetAddress.getAllByName("google.com");
System.out.println("Google có " + addresses.length + " IP addresses:");
for (InetAddress addr : addresses) {
    System.out.println("  - " + addr.getHostAddress());
}
</code></pre>
<p>Kết quả: google.com có 4-6 IP addresses khác nhau! Mình hiểu được cách CDN và load balancing work.</p>
<p><strong>Connectivity check:</strong>
Java có built-in ping functionality:</p>
<pre><code class="language-java">boolean reachable = address.isReachable(5000); // Timeout 5 giây
if (reachable) {
    System.out.println("Server online!");
} else {
    System.out.println("Server offline hoặc network issues");
}
</code></pre>
<p>Trong monitoring app, mình dùng method này để check server health thay vì external ping tools.</p>
<h3>URL - Khi mình học cách parse web addresses</h3>
<p>URL parsing lúc đầu làm mình đau đầu. "https://user:pass@host:port/path?query=value#fragment" - quá nhiều components!</p>
<p>Java URL class giúp parse dễ dàng:</p>
<pre><code class="language-java">import java.net.URL;

URL url = new URL("https://www.example.com:8080/path?param=value#section");

System.out.println("Protocol: " + url.getProtocol());    // https
System.out.println("Host: " + url.getHost());            // www.example.com  
System.out.println("Port: " + url.getPort());            // 8080
System.out.println("Path: " + url.getPath());            // /path
System.out.println("Query: " + url.getQuery());          // param=value
System.out.println("Fragment: " + url.getRef());         // section
</code></pre>
<p>Wow! Từ một string phức tạp, giờ mình có thể extract từng phần riêng biệt. Validation URLs, build dynamic URLs - tất cả trở nên simple.</p>
<h4>Tạo và phân tích URL</h4>
<pre><code class="language-java">import java.net.URL;
import java.net.MalformedURLException;

public class URLExample {
    public static void main(String[] args) {
        try {
            URL url = new URL("https://www.example.com:8080/path/to/resource?param=value#fragment");
            
            System.out.println("Protocol: " + url.getProtocol());
            System.out.println("Host: " + url.getHost());
            System.out.println("Port: " + url.getPort());
            System.out.println("Default Port: " + url.getDefaultPort());
            System.out.println("Path: " + url.getPath());
            System.out.println("Query: " + url.getQuery());
            System.out.println("Fragment: " + url.getRef());
            System.out.println("Full URL: " + url.toString());
        } catch (MalformedURLException e) {
            System.err.println("URL không hợp lệ: " + e.getMessage());
        }
    }
}
</code></pre>
<h3>URI vs URL - Sự khác biệt làm mình confuse</h3>
<p>Lúc đầu, mình không phân biệt được URI và URL. Giáo viên giảng "URI là Uniform Resource Identifier, URL là Uniform Resource Locator" - mình gật gù nhưng không hiểu.</p>
<p><strong>URI - Broad concept:</strong>
URI là định danh tài nguyên, có thể là:</p>
<ul>
<li>URL: https://example.com (có thể truy cập)</li>
<li>URN: urn:isbn:1234567890 (chỉ định danh, không truy cập được)</li>
</ul>
<p>Java URI class flexible hơn URL:</p>
<pre><code class="language-java">URI uri = new URI("https://example.com/path?query=value#fragment");
System.out.println("Scheme: " + uri.getScheme());      // https
System.out.println("Authority: " + uri.getAuthority()); // example.com
System.out.println("Path: " + uri.getPath());          // /path
</code></pre>
<p><strong>URL vs URI trong practice:</strong>
Mình dùng URL khi cần network operations (openStream, openConnection), URI khi chỉ cần manipulate strings.</p>
<p><strong>Real-world application:</strong>
Mình build website availability checker:</p>
<pre><code class="language-java">public static boolean isWebsiteUp(String urlString) {
    try {
        URL url = new URL(urlString);
        HttpURLConnection conn = (HttpURLConnection) url.openConnection();
        conn.setRequestMethod("HEAD");
        conn.setConnectTimeout(5000);
        int responseCode = conn.getResponseCode();
        return responseCode >= 200 &#x26;&#x26; responseCode &#x3C; 400;
    } catch (IOException e) {
        return false;
    }
}
</code></pre>
<p>Trong monitoring system, method này giúp check health của multiple services automatically.</p>
<h2>Những bài học mình rút ra sau project networking</h2>
<p>Sau 2 tháng làm việc với network addresses, mình học được nhiều điều không chỉ về code, mà về real-world networking.</p>
<p><strong>DNS reliability - Không bao giờ trust hoàn toàn:</strong>
DNS có thể fail, timeout, return wrong results. Mình implement caching và fallback mechanisms. Production apps cần handle DNS failures gracefully.</p>
<p><strong>IPv4 vs IPv6 - Compatibility nightmare:</strong>
Java handle cả hai, nhưng logging và storage có thể issues. Mình dùng string representation thay vì binary để avoid compatibility problems.</p>
<p><strong>Network unreliability - Expect failures:</strong>
Mình từng có app crash vì DNS lookup fail. Bây giờ mọi network operation đều có timeout và error handling. Network world unreliable - code phải resilient.</p>
<p><strong>Performance considerations:</strong>
DNS lookup có thể slow (100-500ms). Mình cache results, use connection pooling. Small optimizations impact user experience lớn.</p>
<h2>Kết luận: Network addresses - Gateway to network programming</h2>
<p>Network addresses có vẻ boring lúc đầu, nhưng chúng là foundation của mọi network communication. Từ simple DNS lookup đến complex URL parsing - tất cả đều cần handle properly.</p>
<p>Trong hành trình học Java networking, network addresses dạy mình: network world fragile, validation crucial, error handling mandatory. Đây là mindset quan trọng cho mọi network developer.</p>
<p>Nếu bạn đang học Java networking và thấy network addresses confusing, đừng nản nhé! Mình cũng từng như bạn. Bắt đầu với simple DNS lookup, rồi URL parsing, network monitoring. Practice với real websites, handle edge cases.</p>
<p><em>Lời khuyên cho sinh viên:</em></p>
<ul>
<li>Test với real hostnames (google.com, github.com)</li>
<li>Implement timeout cho mọi network operations</li>
<li>Handle DNS failures gracefully</li>
<li>Learn IPv4/IPv6 differences</li>
</ul>
